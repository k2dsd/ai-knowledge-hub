Phase 01 · 需求发现与机会定义
Problem Discovery & Opportunity Sizing
在投入任何资源之前，明确"AI是否真的是这个问题的最优解"，避免为了AI而AI。
用户研究竞品分析ROI测算 互联网/SaaS制造/工业
▼

核心操作清单

  • 问题界定：用5-Why逐层拆解，明确"是数据问题还是流程问题"
  • 用户访谈：定性访谈20+目标用户，用痛点矩阵排序频率与严重程度
  • 数据可行性评估：预判是否有足量高质训练数据，数据壁垒是否可突破
  • 竞品AI能力地图：绘制竞品AI成熟度（Level 0–4），找差异化切入点
  • 价值主张测试：用Landing Page或纸面原型验证用户是否愿意为AI功能付费
  • 机会规模测算：TAM/SAM/SOM模型，结合AI渗透率假设计算可寻址市场

Phase 02 · 数据战略与数据工程
Data Strategy & Data Engineering
数据是AI产品的"燃料"。数据质量决定模型上限，数据治理决定产品能否规模化。
数据采集数据标注 数据治理特征工程
▼

核心操作清单

  • 数据审计：盘点现有数据资产（量、质、覆盖度、时效性），输出缺口报告
  • 数据采集方案：内部埋点 / 外部爬取 / 第三方购买 / UGC方案选型
  • 标注流水线：设计标注任务SOP、工具选型、质量验收（Kappa系数≥0.8）
  • 数据清洗规范：制定异常值、缺失值、重复数据处理标准，编写检查脚本
  • 特征工程：领域专家+数据科学家协作挖掘强特征，建立特征仓库
  • 数据合规：评估GDPR/个人信息保护法要求，制定脱敏与存储方案
  • 数据飞轮设计：规划用户行为反馈如何自动回流训练管线

Phase 03 · AI模型选型与研发
Model Selection & AI Development
在"自研 vs. 调用API vs. 微调开源模型"之间做出符合商业逻辑的技术决策。
模型评估Fine-tuning RAG架构Prompt工程
▼

核心操作清单

  • 基线方案：先用最简单的规则/统计方法建立Baseline作为比较基准
  • 技术路线决策：自研（数据量大、高度定制）/ API（快速验证）/ 开源微调（平衡成本）
  • 模型评估框架：确定指标（Accuracy/F1/BLEU/NDCG等），建立自动化评估脚本
  • Prompt工程（LLM）：Few-shot、CoT、System Prompt设计与A/B测试
  • RAG架构设计：向量数据库选型、分块策略、检索增强配置
  • 微调策略：LoRA / QLoRA参数高效微调；RLHF对齐用户偏好
  • 安全与伦理：偏见检测、幻觉率测试、对抗样本测试
  • 成本建模：推理成本（Token/请求）建模，计算盈亏平衡点

Phase 04 · 产品设计与人机交互
Product Design & Human-AI Interaction
AI产品UX需解决"不确定性呈现"与"用户信任建立"两大核心挑战。
HAI设计可解释性 原型验证反馈机制
▼

核心操作清单

  • AI能力边界设计：明确AI"能做/不能做/不确定"的三态呈现方案
  • 置信度可视化：将模型置信度转化为用户可理解的界面语言（警告、人工兜底）
  • 反馈闭环设计：设计"点赞/踩、纠错、举报"等反馈组件，构建数据飞轮入口
  • Human-in-the-Loop：界定哪些决策必须人工确认（医疗、金融等高风险场景）
  • 渐进式信任：先展示AI推荐，再逐步减少摩擦（参考Google Maps演进路径）
  • 错误恢复设计：AI出错时的降级方案（Fallback UI），避免用户挫败感
  • 可用性测试：5名用户发现80%可用性问题，重点测试AI输出的理解度

Phase 05 · 工程化与MLOps
Engineering & MLOps
将"能跑的模型"变成"生产级AI系统"，需要完善的MLOps体系支撑持续迭代。
CI/CD模型监控 灰度发布性能优化
▼

核心操作清单

  • 模型服务化：REST API / gRPC封装，定义输入输出Schema，版本化管理
  • 推理优化：量化（INT8/FP16）、蒸馏、ONNX转换，降低延迟与推理成本
  • CI/CD for ML：代码变更自动触发模型重训练、评估、A/B测试流水线
  • 数据漂移监控：实时检测输入数据分布变化（Data Drift），触发重训警告
  • 模型衰减监控：追踪线上关键指标（精确率/召回率/业务KPI），设置告警
  • 灰度发布：Canary Release（5%→20%→50%→100%），配合指标验证
  • 降级兜底：模型超时/异常时自动切换至规则引擎或上一版本

MLOps成熟度等级

L0 · 手动流程 — 脚本手跑，无版本管理，不可重现
L1 · ML Pipeline — 训练流水线自动化，模型版本管理
L2 · CI/CD for ML — 代码→训练→评估→部署全自动
L3 · 自适应系统 — 数据漂移自动触发重训，持续学习

Phase 06 · 测试、评估与安全上线
Testing, Evaluation & Safe Launch
AI产品测试远比传统软件复杂——需同时验证技术指标、用户体验和业务价值。
A/B测试离线评估 安全审查压力测试
▼

核心操作清单

  • 离线评估：在黄金测试集上验证核心模型指标，设定上线准入门槛
  • Shadow Mode：新旧模型并行运行对比输出差异，不影响线上用户
  • A/B实验设计：明确假设、最小样本量（统计显著性）、主指标/护栏指标
  • Red Team测试（LLM）：专项测试幻觉、越狱、有害内容、隐私泄露
  • 负向案例库：收集并测试100+边缘案例，确保极端输入下行为可控
  • 性能压测：模拟峰值流量，验证P99延迟、QPS、服务稳定性
  • 回滚预案：制定回滚触发条件，确保5分钟内可回滚至上一版本

三层评估框架

🔬 技术层指标

Accuracy / F1 / Latency / Throughput / Memory

👤 用户体验层

任务完成率 / 满意度NPS / 功能使用率 / 报错率

💼 业务层指标

转化率 / 留存率 / ROI / 成本节约 / 收入增量

Phase 07 · 商业化运营与持续迭代
Commercialization & Continuous Iteration
AI产品的价值在上线后才真正开始创造，构建持续学习与优化的正向飞轮。
数据飞轮定价策略 持续学习规模化
▼

核心操作清单

  • GTM策略：Beta用户招募、分层灰度、使用激励方案设计
  • AI功能定价：基于价值定价（WTP评估），而非成本加成定价
  • 数据飞轮运营：分析用户纠错/反馈数据，每2–4周回流至训练管线
  • AI解释性运营：向用户/监管方解释AI决策依据（金融/医疗合规要求）
  • 客户成功：企业客户配合落地培训，制定"AI效益量化报告"增强续费
  • 规模化策略：多租户隔离、私有化部署选项、API扩展能力建设
  • 版本迭代节奏：区分"模型更新"与"产品更新"的发布节奏管理

AI产品定价模型

用量计费 — 按Token/API调用/分析次数（适合LLM、分析类产品）
价值分成 — 按AI带来的成本节约或收入增量分成（适合工业场景）
席位订阅 — 按用户数/席位收取固定月费（适合协作类SaaS）

AOI视觉缺陷检测系统
核心问题
PCB板与电子元器件外观质检依赖人工，漏检率约0.3–1%，人力成本高，检测员疲劳导致质量波动。

MindSphere · 工业AI预测性维护
核心问题
工业设备（电机、压缩机、CNC机床）突发故障造成的非计划停机，一次停机损失可达数十万至数百万元。
技术路线
从设备传感器（振动、温度、电流、转速）采集时序数据，使用LSTM时序模型 + Isolation Forest异常检测构建设备健康评分（0–100），在故障发生前3–7天触发维护预警，集成数字孪生模拟故障传播路径。
工作流关键决策
① 修复性→预防性→预测性三级演进路径，先跑通数据链路再上AI；② 与客户共同定义"健康评分阈值"确保工程师信任告警；③ 基于"避免一次停机"价值分成定价。

FANUC · Preferred Networks
ZDT · 机器人零停机系统
核心问题
工业机器人（尤其汽车焊接线）突发故障导致整条产线停机，停线1小时损失可达数十万元。
技术路线
FANUC联合Preferred Networks构建ZDT云平台：将全球数万台FANUC机器人运行数据汇入云端，通过联邦学习在不暴露原始数据的前提下训练统一故障预测模型（LSTM + 自编码器异常检测）。
工作流关键决策
① 联邦学习解决数据孤岛：数据不出厂，共享模型梯度，FANUC获得全量训练效果；② 接入机器人越多模型越准，形成强数据网络效应壁垒；③ ZDT订阅与机器人销售捆绑，形成高毛利服务收入。

Anti-Patterns

AI产品研发 · 高频陷阱与应对

基于真实项目总结的常见失败模式，每一条都有对应解法

🎯
为AI而AI，不从业务问题出发
团队为追赶潮流强行加入AI功能，但该场景本质是简单规则问题，AI带来额外复杂度却没有额外价值。典型案例：用ML预测"按钮颜色对转化率的影响"，A/B测试足以解决。
✅ 应对：先建立规则Baseline，仅在规则无法解决、数据量足够、人工成本明显高于AI成本时才引入AI。
📊
忽视数据质量，过度关注模型算法
"Garbage In, Garbage Out"是AI领域最古老的铁律，却仍是最常见的失败原因。花80%时间调模型，只花20%时间做数据清洗，结果模型在测试集好但线上一塌糊涂。
✅ 应对：数据清洗与特征工程通常占项目70%工时，这是正常比例。宁可花2周清洗数据，不要花2个月调模型。
📈
只关注模型指标，不关注业务指标
Accuracy提升了2%，但业务核心指标（转化率/GMV/良率）没有变化。模型优化和产品价值之间存在断层，无法说服业务方和高层继续投入。
✅ 应对：立项时定义"北极星业务指标"，每次模型迭代必须说明预计对业务指标的影响量级，建立技术→业务指标映射关系。
🚀
跳过MVP直接做大而全系统
在需求未充分验证的情况下，耗费6–12个月构建完整AI系统，上线后发现用户根本不买账，或核心假设有误，造成大量浪费。（制造业项目尤其常见）
✅ 应对：先用最简单的方案验证产品假设，4–6周快速出POC，核心假设经用户验证后再投入规模化研发。
🔒
低估数据合规与安全风险
训练数据包含用户隐私信息却未脱敏，或LLM产品未做越狱防护，上线后遭遇合规处罚或公关危机。2023年多家企业因将用户数据输入ChatGPT API导致数据泄露。
✅ 应对：立项阶段引入法务/合规评审；LLM产品必须配置内容过滤、PII识别脱敏、用户数据训练Opt-out等基础安全措施。
👻
上线即终点，缺乏模型维护体系
AI产品上线后团队就解散，没有监控体系、没有定期重训计划。模型因数据漂移逐渐"腐烂"，6个月后性能大幅下降，但团队毫不知情。
✅ 应对：将"模型监控与维护"纳入研发交付物，明确重训触发条件、负责团队、预算预留（通常为首年开发成本的20–30%）。
