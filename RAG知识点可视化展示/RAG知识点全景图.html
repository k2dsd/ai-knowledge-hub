什么是 RAG

为什么每个 AI 产品都需要 RAG？

检索增强生成（Retrieval-Augmented Generation）让大模型能够访问实时、权威的外部知识，解决知识过时、幻觉和私域数据访问三大核心问题。

🧠

解决知识截止问题

LLM 训练数据有截止日期，无法获取最新信息。RAG 通过实时检索知识库，让模型回答基于最新、最准确的数据。

✅

大幅降低幻觉率

纯 LLM 幻觉率普遍在 15-30%，引入 RAG 后可降至 3-8%。模型生成答案有据可查，每个陈述都可溯源到原文。

🔒

访问私域数据

企业内部文档、合同、知识库无需上传到模型训练，只需构建向量索引即可让 AI 安全访问，数据主权归企业所有。

⚡

比微调更经济灵活

微调成本高昂且难以快速更新。RAG 知识库随时更新，无需重新训练模型，总体成本降低 70-90%。

📎

引用与溯源

RAG 系统可在回答中附上原始文档来源，用户可验证每条信息出处，在法律、医疗、金融等高要求场景不可或缺。

📈

市场规模爆发

据预测，RAG 市场 2024 年约 13 亿美元，2029 年将达 98 亿美元，CAGR 超 49%。金融、医疗、法律是最大驱动行业。

文本分块策略对比

分块是 RAG 系统最关键的预处理步骤之一，直接影响检索精度。

策略原理优点缺点推荐场景
固定大小分块按 Token 数截断实现简单、速度快可能割裂语义均匀短文本
递归字符分块 ⭐按层级分隔符递归切分平衡大小与完整性需调优分隔符通用场景（最常用）
语义分块按语义相似度聚合句子语义完整块大小不均长文报告
父子分块 ⭐小块检索 + 大块传 LLM精准检索+完整上下文索引复杂高精度场景
文档结构分块利用 MD/HTML 结构保持文档层级依赖文档格式结构化文档
滑动窗口分块带重叠的滑动窗口减少边界信息丢失冗余数据增多连续性强文本
检索策略

选择正确的检索策略
决定 RAG 系统上限

检索质量直接决定 RAG 系统的上限。根据业务场景特点进行策略选择，是 AI 产品设计的核心决策。

稀疏检索

BM25 / TF-IDF

基于词频统计的经典信息检索算法。关键词精确匹配，延迟极低，适合专业术语密集场景。

✅ 优势延迟低、无需 GPU、关键词精准、可解释性强
⚠️ 劣势无法理解语义相似性、同义词匹配弱
密集检索

向量语义检索

用 Embedding 模型将文本编码为向量，通过 ANN 检索语义相近文档。理解语义比字面更准确。

✅ 优势理解语义、同义词鲁棒、跨语言支持好
⚠️ 劣势需要 GPU、精确关键词匹配差
混合检索 ⭐

Hybrid Search

结合 BM25 和向量检索双路召回，通过 RRF（倒数排名融合）合并结果。企业场景鲁棒性最强。

✅ 优势召回全面、互补优缺点、企业级首选
⚠️ 劣势架构略复杂、融合参数需调优
图谱检索

Graph-based Retrieval

将知识结构化为实体关系图谱，通过图遍历检索多跳关联知识，适合需要推理的复杂问答。

✅ 优势多跳推理、关联分析、结构化领域
⚠️ 劣势图谱构建成本高、维护复杂
重排序

Reranker（Cross-encoder）

检索后对候选文档精细打分排序。BGE-Reranker / Cohere Rerank 可使 Top-K 精准率提升 15-25%。

✅ 优势精准率大幅提升、与召回解耦
⚠️ 劣势增加 50-200ms 延迟
查询增强

Query Enhancement

包括查询改写、HyDE（假设文档生成）、多查询扩展、查询分解等技术，提升检索召回率。

✅ 优势低成本提升召回率、对齐语义
⚠️ 劣势增加 LLM 调用次数和延迟
技术演进

从 Naive 到 Agentic
RAG 五代技术范式

RAG 技术持续演进，每一代都在解决前一代的核心瓶颈。了解技术路线，帮助产品团队做出正确的技术选型。

基础范式 · 2022

Naive RAG

最基础实现：固定分块 → 向量化 → 单路检索 → 直接生成。适合 POC 验证和简单问答场景。

  • 固定大小分块 + 单路向量检索
  • Top-K 直接塞入 Prompt
  • 无重排序、无查询优化
  • 适合快速 POC 验证
进阶范式 · 2023

Advanced RAG

在检索前、检索中、检索后多个环节引入优化，系统性提升质量，是当前企业落地主流方案。

  • 查询改写 / HyDE / 多查询扩展
  • 父子分块 + 混合检索
  • Cross-encoder 重排序
  • 上下文压缩（Contextual Compression）
模块化 · 2023

Modular RAG

将 RAG 各模块完全解耦，可独立替换和组合，形成积木式灵活架构，适合大规模企业部署。

  • 检索模块可插拔替换
  • 支持多种检索模式动态切换
  • RAGAs 框架独立评估各模块
  • 适合大规模企业级平台
图增强 · 2024

GraphRAG

微软 2024 年开源。构建文档实体关系图谱，支持多跳推理和社区摘要，全局查询能力大幅超越传统 RAG。

  • 自动抽取实体关系图谱
  • 社区检测算法聚合知识
  • 局部查询 + 全局查询双模式
  • 金融、医疗知识图谱效果显著
智能体 · 2024-25

Agentic RAG

LLM 作为 Agent 自主规划检索策略，动态决策何时检索、检索什么，形成反思自我校验闭环。

  • Self-RAG：生成时自判断是否需检索
  • CRAG：检索后评估文档相关性
  • ReAct + RAG：工具调用 + 推理
  • 多 Agent 协作分工检索
多模态 · 2024-25

Multimodal RAG

支持图片、表格、视频等多模态内容检索与生成，应对企业文档中大量图文混排的实际情况。

  • ColPali：文档图像直接向量化
  • GPT-4V / LLaVA 多模态生成
  • 图表理解与表格问答
  • PDF 图像内容跨模态检索

评估体系

科学评估持续优化 RAG 系统

没有评估就没有优化。从检索质量和生成质量两个维度全面衡量系统表现，建立量化迭代基线。

检索侧指标

🎯

Context Precision

Top-K 文档中，真正相关的比例。衡量「检回来的准不准」。

重要性核心指标
📡

Context Recall

答案所需的关键信息，是否都被检索到了。衡量「有没有漏掉」。

重要性核心指标
🔗

Context Relevance

检索文档与用户问题的语义相关程度，通常用 LLM 打分评估。

重要性重要
📊

MRR / NDCG

MRR（平均倒数排名）和 NDCG 衡量检索排序质量，越高越好。

重要性重要

生成侧指标

✅

Faithfulness（忠实度）

生成陈述是否都能从上下文中找到支撑。衡量「有没有幻觉」，核心中的核心。

重要性最高优先级
💡

Answer Relevance

生成答案是否真正回答了用户的问题，衡量有效性和针对性。

重要性核心指标
🌡️

Hallucination Rate

答案中包含上下文不存在内容的比例。目标：越低越好，企业要求低于 5%。

重要性最高优先级
🏁

End-to-End Accuracy

最终答案与标准答案的匹配度（精确匹配、F1、BERTScore 等方式评估）。

重要性重要

主流评估框架

🔬

RAGAs（最推荐）

最流行的 RAG 开源评估框架。提供 Faithfulness、Answer Relevance、Context Precision/Recall 全套指标，支持无参考答案的自动化评估，可与 LangChain / LlamaIndex 无缝集成。

📏

TruLens

ARES（斯坦福）用 LLM 自动生成评估数据集；BEIR 是信息检索领域标准 Benchmark，含 18 个多样化数据集，用于评估检索模型零样本泛化能力。

工程落地

最佳实践与落地路线图

来自真实项目的经验总结，帮助 AI 产品经理在产品设计和技术对齐中做出正确决策。

📥 数据与索引阶段

🔧

文档解析质量是一切的基础

▼

60% 的 RAG 问题出在数据质量上。关键实践：

  • PDF 优先使用 PyMuPDF + 版式分析，避免直接 OCR 扫描件
  • 表格单独提取为 Markdown / JSON，不要作纯文本处理
  • 删除页眉、页脚、页码等噪声内容
  • 使用 Docling（IBM 开源）或 LlamaParse 处理复杂排版
✂️

分块策略直接影响召回精度

▼
  • 中文 200-400 Token，英文 400-600 Token，重叠率 10-20%
  • 优先使用递归字符分块（RecursiveCharacterTextSplitter）
  • FAQ 类文档直接以问答对作为最小索引单元
  • 父子分块：小块检索 + 大块传给 LLM
🏷️

元数据设计决定过滤效率

▼
  • 每个文档块添加来源、日期、作者、部门、版本元数据
  • 支持按元数据过滤，避免过期文档干扰
  • 文档更新时设置版本控制和失效机制

🔍 检索与生成阶段

🎯

混合检索 + 重排序是标准配置

▼
  • 生产环境务必使用 BM25 + 向量混合检索
  • RRF（Reciprocal Rank Fusion）融合，α=0.5 作为默认起点
  • 加入 BGE-Reranker，Top-K 精准率提升 15-25%
  • Reranker 只对 Top-20 候选排序，控制延迟
📝

Prompt 工程是低成本优化杠杆

▼
  • System Prompt 明确约束「只基于提供的上下文回答」
  • 检索文档加编号和来源标注，方便引用追溯
  • 设置置信度阈值，低置信度时回复「知识库中暂无相关信息」
  • Few-shot 示例展示期望的答案格式
🚀

性能优化与工程架构

▼
  • 向量检索使用 HNSW 索引，召回速度毫秒级
  • Semantic Cache 缓存高频问题，降低 60%+ API 成本
  • 异步检索 + 流式输出，提升用户感知响应速度
  • 多租户场景使用命名空间隔离，确保数据安全

避坑指南

8 大常见问题与解决方案

从真实项目踩坑经验中提炼，帮助团队避开 RAG 工程落地中的高频雷区。

⚠ 检索相关但答案仍错误

检索到相关文档，但 LLM 在生成时产生幻觉，添加了上下文中不存在的内容。

✅ 解决方案System Prompt 强制约束只基于上下文回答；加入 Faithfulness 实时评估；使用 Self-RAG 让模型自我校验引用。

⚠ 语义相似但内容不相关

向量检索召回语义相似但实际无关的文档，如「苹果」匹配到水果而非苹果公司。

✅ 解决方案引入元数据过滤（按领域/时间）；使用 Reranker 精细打分；加 BM25 互补精确关键词匹配。

⚠ 知识库更新不及时

文档更新后向量库未同步，在政策法规、价格等敏感信息上风险极高。

✅ 解决方案设计文档变更触发的自动更新流水线；加入文档时效性元数据；对时间敏感查询强制过滤最新版本。

⚠ 长文档语义被切断

固定大小分块导致完整论点被截断在不同 Chunk 中，单独任何一块都不完整。

✅ 解决方案使用语义分块或按段落边界切分；父子分块策略；加入 Overlap 重叠区域（15-20%）。

⚠ 多文档信息冲突

检索到多来源文档包含相互矛盾信息（如不同版本规定），LLM 难以判断采信哪个。

✅ 解决方案文档加版本/日期元数据并按优先级排序；Prompt 指示优先采信最新/官方来源。

⚠ 延迟过高影响体验

检索 + 重排序 + LLM 推理总延迟超过 5 秒，特别在移动端和实时客服场景。

✅ 解决方案流式输出优先响应；Semantic Cache 缓存高频问题；异步预检索；选用更快的 LLM（Haiku/Flash）。

⚠ 多语言跨语言检索

用户中文提问，知识库中有大量英文文档，普通 Embedding 跨语言效果差。

✅ 解决方案使用多语言 Embedding（BGE-M3、multilingual-e5）；对查询进行语言检测，必要时翻译后检索。

⚠ 权限管控缺失

不同角色应看到不同范围内容（员工/经理/HR），但 RAG 统一全库检索造成越权。

✅ 解决方案向量库使用 Namespace 或元数据标签按用户角色过滤；检索时注入用户权限上下文；定期审计。
